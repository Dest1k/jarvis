# Матрица распределения VRAM — RTX 5090 (32 ГБ)

Документ детально обосновывает выбор флагов `--gpu-memory-utilization` и
жёсткий резерв памяти под хост-операции.

## Исходные данные

- Полный объём VRAM RTX 5090: **32.0 ГБ**
- Требуемый резерв хоста (Windows + Xvfb/виртуальный фреймбуфер + пул KV-кэша): **~5.5 ГБ**
- Доступно под инференс: **32.0 − 5.5 = 26.5 ГБ**

## Распределение

| Сервис | Модель | Квантизация | Бюджет VRAM | Флаг |
|--------|--------|-------------|-------------|------|
| Диспетчер + Кодер | Qwen2.5-Coder-32B-Instruct | Int4 AWQ (Marlin) | **19.0 ГБ** | `--gpu-memory-utilization 0.60` |
| Контроллер ОС/GUI | UI-TARS-7B | Int4 | **5.5 ГБ** | `--gpu-memory-utilization 0.17` |
| Аудио (ASR+TTS) | Faster-Whisper L-v3 + Kokoro | int8_float16 / FP16 | **2.0 ГБ** | — (не vLLM) |
| **Резерв хоста** | — | — | **5.5 ГБ** | hardcoded |
| **ИТОГО** | | | **32.0 ГБ** | |

## Математика `--gpu-memory-utilization`

`gpu_memory_utilization` в vLLM — это доля **полной** памяти устройства,
которую процессу разрешено зарезервировать под веса + активации + KV-кэш.
При совместном размещении нескольких инстансов на одном GPU:

1. Доли **суммируются** и не должны в сумме давать перерасход.
2. Инстансы поднимаются **последовательно** (`depends_on` в compose):
   каждый следующий измеряет уже уменьшенный объём свободной памяти.

```
Шаг 1 — Qwen:    0.60 × 32 = 19.2 ГБ  →  занято 19.2,  свободно 12.8
Шаг 2 — UI-TARS: 0.17 × 32 =  5.4 ГБ  →  занято 24.6,  свободно  7.4
Шаг 3 — Audio (CTranslate2/torch, вне vLLM): ~2.0 ГБ → занято 26.6, свободно 5.4
                                                          └──────── резерв ≈ 5.4 ГБ
```

Итоговый headroom ≈ **5.4 ГБ** удовлетворяет требованию ~5.5 ГБ
(с учётом фрагментации и пиков KV-кэша при `--max-model-len 32768`).

## Контекст и батчинг

- Qwen-Coder: `--max-model-len 32768`, `--max-num-seqs 16`, `--enable-prefix-caching`.
  Длинный контекст требует значимого KV-кэша; именно поэтому Qwen получает
  максимальную долю (0.60), а резерв сознательно завышен.
- UI-TARS: `--max-model-len 8192`, `--max-num-seqs 4` — модели хватает короткого
  контекста для парсинга состояния экрана.

## Проверочные команды

```bash
# Текущая занятость VRAM по процессам
nvidia-smi --query-compute-apps=pid,used_memory --format=csv

# Здоровье инстансов vLLM
curl -s http://localhost:8001/health && echo "  Qwen OK"
curl -s http://localhost:8002/health && echo "  UI-TARS OK"
curl -s http://localhost:8003/health && echo "  Audio OK"

# Полная карта памяти
nvidia-smi
```

## Тюнинг при OOM

Если при старте возникает `CUDA out of memory`:

1. Снизьте `JARVIS_QWEN_GPU_UTIL` с `0.60` до `0.55` (Qwen → ~17.6 ГБ).
2. Уменьшите `--max-model-len` Qwen до `16384` (меньше KV-кэша).
3. Снизьте `--max-num-seqs` до `8`.
4. Для UI-TARS можно опустить `0.17` → `0.15`.
