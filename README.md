# JARVIS-OS

**Самособирающаяся локальная мультиагентная система** для Windows 11 (WSL2 + vLLM + Docker).

> ⚠️ **Статус артефактов.** Это связный референс-каркас production-уровня с рабочей логикой,
> предназначенный для запуска и адаптации на реальном целевом железе. Перед боевым
> использованием обязательно проверьте каждый компонент в вашем окружении. Система выполняет
> привилегированные операции на хост-машине Windows — внимательно изучите раздел «Безопасность».

## 3. Матрица распределения VRAM (RTX 5090 — 32 ГБ)

Стратегия: жёстко зарезервировать **~5.5 ГБ** под хост Windows, виртуальный фреймбуфер (Xvfb)
и динамический пул KV-кэша. Под vLLM/инференс остаётся **26.5 ГБ**.

| # | Сервис | Модель | Квантизация | VRAM | `--gpu-memory-utilization` | Примечание |
|---|--------|--------|-------------|------|----------------------------|-----------|
| 1 | Dispatcher / Coder | Qwen2.5-Coder-32B-Instruct | Int4 AWQ | **~19.0 ГБ** | `0.60` | `--max-model-len 32768` |
| 2 | OS / GUI Controller | UI-TARS-7B | Int4 | **~5.5 ГБ** | `0.17` | парсинг состояний экрана |
| 3 | Audio / Voice | Faster-Whisper Large-v3 + Kokoro TTS | FP16/Int8 | **~2.0 ГБ** | — (не vLLM) | реальное время |
| — | **Резерв хоста** | Windows + Xvfb + KV-pool | — | **~5.5 ГБ** | — | hardcoded headroom |
| | **ИТОГО** | | | **32.0 ГБ** | | |

### Почему именно такие `--gpu-memory-utilization`

`gpu_memory_utilization` в vLLM — это доля **полной** памяти устройства, которую процессу
разрешено занять (веса + активации + KV-кэш). При совместном размещении нескольких инстансов
на одном GPU доли **суммируются** и не должны перекрываться, а инстансы поднимаются
**последовательно** (каждый следующий видит меньше свободной памяти):

```
Инстанс 1 (Qwen):    0.60 × 32 ГБ = 19.2 ГБ   →  занято 19.2 ГБ,  свободно 12.8 ГБ
Инстанс 2 (UI-TARS): 0.17 × 32 ГБ =  5.4 ГБ   →  занято 24.6 ГБ,  свободно  7.4 ГБ
Audio (CTranslate2/torch, не vLLM):  ~2.0 ГБ  →  занято 26.6 ГБ,  свободно  5.4 ГБ  ← резерв
```

Итоговый headroom ≈ **5.4 ГБ** соответствует требованию ~5.5 ГБ.

## 5. Порядок запуска (полностью автоматический)

```powershell
# 1. На ХОСТЕ Windows (PowerShell, нативный Python 3.11+)
pip install requests
python bootstrap_installer.py --lmstudio http://localhost:1234/v1 --wsl-ram 96 --wsl-cpus 20

# 2. Демон RPC-моста (отдельное окно, на хосте)
pip install websockets
python windows_rpc_bridge.py --port 8765

# 3. Дашборд
cd dashboard && npm install && npm run dev   # http://localhost:3000
```

> **ЕДИНАЯ МОДЕЛЬ.** Весь путь установки использует ОДНУ локальную модель.
> `bootstrap_installer.py` по умолчанию **НЕ обращается к LM Studio** и не
> загружает никакую модель (профиль детерминированный, корректен для RTX 5090) —
> иначе вместе с моделью агента-установщика это вызывает **OOM** на 32 ГБ GPU.
> «Мозг» установки — `install_agent.py` (§5.2), работающий на одной модели.
> Запрос профиля у LLM можно включить явно: `--use-lmstudio --model <id>` (тот же
> id, что у агента).

`bootstrap_installer.py` выполняет ВСЁ автоматически, без ручных шагов:

1. Проверяет хост (Windows / WSL2 / Docker Desktop / драйвер NVIDIA).
2. Формирует профиль развёртывания **детерминированно** (без обращения к LLM).
3. **Автоматически определяет имя установленного WSL-дистрибутива**, а при его
   отсутствии — ставит Ubuntu (`--no-auto-install` отключает).
4. Пишет оптимальный `.wslconfig`.
5. **Автоматически доводит поддержку GPU в контейнерах** (Docker Desktop «из
   коробки» либо авто-установка NVIDIA Container Toolkit для нативного docker).
6. Генерирует `wsl/.env` и **поднимает весь стек** через хостовый Docker Desktop
   (`docker compose up -d`), дожидаясь готовности сервисов.

> Тест GPU и подъём стека идут через **хостовый движок Docker Desktop**, поэтому
> не зависят от конкретного имени WSL-дистрибутива. Альтернативный WSL-native
> сценарий — `wsl/wsl_setup_orchestrator.sh` (на случай нативного docker без
> Docker Desktop).

Полезные флаги: `--model <id>` (модель LM Studio), `--distro <name>` (дистрибутив),
`--skip-stack` (только подготовка), `--skip-prefetch` (не предзагружать веса),
`--qwen-model` / `--uitars-model` (веса агентов).

### Кеширование, возобновляемость и повторяемость

Процесс рассчитан на обрывы сети и повторный запуск **без вмешательства** —
просто запустите ту же команду ещё раз, и она продолжит с места обрыва:

| Что | Кешируется / возобновляется |
|-----|------------------------------|
| **Веса моделей** (~19 ГБ Qwen и др.) | Предзагрузка в персистентный кэш `D:\jarvis\data\hf` с **докачкой** (`hf_transfer` выключен → resume) и ретраями (6 попыток). Уже скачанное **не качается заново**. |
| **Docker-образы** | Послойный кэш Docker: завершённые слои не перекачиваются. `pull/build` обёрнуты в ретраи (4 попытки). |
| **Перенос на D:** | Идемпотентен: проект/дистрибутив/Docker, уже находящиеся на D:, пропускаются. |
| **Сборка `build`** | Кэш слоёв Docker: неизменные шаги (apt/pip) берутся из кэша. |

Ключевая причина прежних «повторных закачек одного и того же» — в образе vLLM
включён `hf_transfer`, который **не умеет докачку** и при обрыве качал файл с нуля.
Теперь `HF_HUB_ENABLE_HF_TRANSFER=0` (и для предзагрузки, и для рантайма vLLM/аудио),
а кэш HF лежит на постоянном bind-mount → загрузка **возобновляется**.

> Веса моделей кладутся на D: через **bind-mount** (`D:\jarvis\data\hf`) и не
> зависят от того, где расположено хранилище образов Docker.

### 5.1 Перенос на диск D: (`D:\jarvis`)

На старте (ФАЗА 0) bootstrap переносит проект и весь «тяжеляк» на целевой диск,
после чего всё работает уже оттуда:

| Что переносится | Куда | Как |
|-----------------|------|-----|
| Файлы проекта | `D:\jarvis` | копирование + переключение рабочей директории |
| Веса моделей / кэш HF / sandbox (~25+ ГБ) | `D:\jarvis\data\{models,hf,sandbox}` | bind-mount в compose (`JARVIS_DATA_DIR`) |
| WSL-дистрибутив (образ ВМ) | `D:\jarvis\wsl\distro` | `wsl --export → --unregister → --import` (с архивом-резервом) |
| Диск Docker Desktop (образы) | `D:\jarvis\docker` | смена «Disk image location» **+ фактический перенос распределения `docker-desktop-data`** (export/import) + перезапуск и верификация |
| Исходная папка на C: | удаляется | после успешного переноса |

```powershell
# Перенос включён по умолчанию (целевой путь D:\jarvis)
python bootstrap_installer.py --lmstudio http://localhost:1234/v1

# Тонкая настройка переноса:
#   --target-root E:\jarvis   другой диск/путь
#   --no-relocate             не переносить (работать на месте)
#   --no-move-docker          не трогать диск Docker Desktop
#   --no-move-distro          не переносить WSL-дистрибутив
#   --keep-source             оставить копию проекта на C:
```

> ⚠️ Перенос диска Docker Desktop и WSL-дистрибутива — Windows-специфичные,
> версионно-зависимые операции. Bootstrap делает их защищённо (бэкап настроек
> Docker, экспорт дистрибутива перед `--unregister`, пропуск без аварии при
> ошибке), но первый запуск стоит наблюдать.

**Префлайт-страховка от утечки образов на C:.** Перед `docker compose pull/up`
bootstrap определяет РЕАЛЬНОЕ расположение данных Docker через реестр WSL
(`HKCU\…\Lxss` → `BasePath` дистрибутивов `docker-desktop-data`/`docker-desktop`)
и сравнивает диск с целевым. Если данные Docker всё ещё **не на `D:`** —
**останавливается** с инструкцией (чтобы образы ~10+ ГБ не утекли на C:).
Обойти осознанно: `--allow-docker-on-c`.

**«Если что-то уже живёт на D: — не трогаем».** Перенос идемпотентен и
неразрушающий:
- файлы проекта, **уже существующие на D:, не перезаписываются** (копируются
  только недостающие); для принудительного обновления — флаг `--overwrite-existing`;
- WSL-дистрибутив с маркером `.relocated` повторно не экспортируется;
- Docker, уже расположенный на D:, повторно не переносится;
- если при копировании были ошибки — исходная папка на C: **не удаляется**
  (чтобы не потерять не скопированное).

## 5.2 Единый автономный установщик на ОДНОЙ модели (`install_agent.py`)

**Рекомендуемый путь.** Полностью самостоятельный режим: ОДНА локальная LLM
(LM Studio) ведёт ВЕСЬ процесс установки — анализирует состояние хоста,
принимает решения, исполняет команды, диагностирует ошибки и восстанавливается,
**без вмешательства**.

```powershell
pip install requests
python install_agent.py --lmstudio http://localhost:1234/v1 --target-root D:\jarvis
```

> **Одна модель на всё.** Агент использует единственную модель (по умолчанию
> `gemma-4-31b-it-qat`; переопределяется `--model <id>`). Его
> инструмент `run_bootstrap` вызывает детерминированный `bootstrap_installer.py`,
> который **не загружает вторую модель**, а агент дополнительно вырезает любые
> флаги загрузки модели — поэтому **OOM из-за двух моделей исключён**.
> В LM Studio должна быть загружена только эта одна модель.

**Два режима (устойчивость к «тупящим» моделям):**
- **Управляемый (по умолчанию).** Счастливый путь идёт детерминированным планом
  (get_state → run_bootstrap → RPC-мост → npm install → дашборд → ожидание
  готовности → отчёт). Модель подключается **только для диагностики и починки
  сбоев**. Поэтому даже слабая модель (например `qwen3-coder-30b-a3b` —
  MoE с ~3B активных параметров) доводит установку до конца.
- **Свободный** (`--free-agent`). Полный ReAct-цикл: модель сама решает каждый
  шаг. Подходит для сильных моделей.

Агент доводит систему до конца — не только docker-стек, но и хостовые сервисы:

| Слой | Сервис | Порт |
|------|--------|------|
| Docker | backend (FastAPI + LangGraph) | 8000 |
| Docker | vLLM Qwen2.5-Coder-32B (диспетчер+кодер) | 8001 |
| Docker | vLLM UI-TARS-7B (контроллер ОС/GUI) | 8002 |
| Docker | audio (Faster-Whisper + Kokoro) | 8003 |
| Docker | sandbox (изолированное исполнение кода) | — |
| Хост | windows_rpc_bridge.py (мост + HITL) | 8765 |
| Хост | dashboard (Next.js 15 Command Center) | 3000 |

**Критерий готовности:** `check_endpoints` → `_ГОТОВО_ПОЛНОСТЬЮ=ДА` (все порты
выше отвечают). Только тогда агент вызывает `finish`.

**Финал.** При завершении агент автоматически: снимает итоговый статус всех
сервисов, пишет отчёт `D:\jarvis\logs\deployment_report_*.md` (дата, модель,
состояние сервисов, точки доступа) и **открывает дашборд** `http://localhost:3000`
в браузере.

**Инструменты агента:** `get_state`, `run_powershell`, `run_cmd`, `run_wsl`,
`docker`, `read_file`, `write_file`, `http_get`, `start_background` (хостовые
демоны: RPC-мост, dev-сервер дашборда), `check_endpoints` (проверка всех
сервисов), `run_bootstrap` (делегирование проверенному `bootstrap_installer.py`),
`finish`.

**Безопасность (защита от галлюцинаций модели):**
- **Катастрофические** команды (формат/очистка диска, `diskpart`, `rm -rf` корня,
  `mkfs`, удаление системного реестра, `bcdedit`) — **жёстко блокируются всегда**.
- **Деструктивные** (`del`, `taskkill`, `git push`, `docker rmi`, `wsl --unregister`) —
  в авто-режиме выполняются сами; `--require-approval` переводит их в запрос
  подтверждения в консоли.
- Ограничение шагов (`--max-steps`), детектор зацикливания, `--dry-run`
  (показывать намерения без исполнения), полный транскрипт в `D:\jarvis\logs\`.

> **Кодировка консоли.** Все хостовые скрипты при старте переключают консоль
> Windows в UTF-8 (`SetConsoleOutputCP(65001)` + `reconfigure`), чтобы кириллица
> не превращалась в «ромбики со знаком вопроса».

> **Живой прогресс.** Длинные шаги (развёртывание через `run_bootstrap`,
> `npm install`) выводятся в реальном времени: bootstrap запускается с `-u`
> (небуферизованно), а вывод читается сырыми чанками (`read1`), поэтому
> прогресс-бары `docker pull` и загрузки весов HuggingFace видно вживую, а не
> «в тишину» до завершения.

| Флаг | Назначение |
|------|-----------|
| `--model <id>` | модель LM Studio (по умолчанию `gemma-4-31b-it-qat`) |
| `--free-agent` | свободный ReAct-режим (по умолчанию — управляемый) |
| `--require-approval` | спрашивать перед деструктивными командами |
| `--dry-run` | не исполнять, только показывать намерения |
| `--max-steps N` | лимит шагов свободного режима (по умолчанию 80) |
| `--goal "..."` | дополнительное уточнение задачи |

> Агент опирается на `run_bootstrap`, поэтому получает «бесплатно» всю
> проверенную логику: перенос на D:, префлайт Docker, возобновляемую загрузку
> весов, идемпотентность.

## 6. Безопасность

- **Токен-хендшейк.** RPC-мост принимает команды только при совпадении одноразового токена,
  записанного в `~/.jarvis/bridge.token` (права `600`). Слушает исключительно `127.0.0.1`.
- **HITL-гейт.** Любая команда из чёрного списка (удаление, форматирование, `git push`,
  сетевые операции, запуск процессов) **останавливает исполнение** и ждёт визуального
  подтверждения в дашборде.
- **Git-автоматика.** Целевая ветка задаётся `JARVIS_GIT_BRANCH` (по умолчанию
  `jarvis/auto-updates`, **не** `main`). Пуш в `main` запрещён без явного флага оператора —
  чтобы ИИ-сгенерированный код не попадал в основную ветку без ревью.
- **Изоляция кода.** Кодер-агент исполняет код только в dockerized-sandbox без проброса
  хост-ФС за пределы рабочей директории задачи.
